Implementation Plan

BodyBuddy Core v0.1 - Phased Development Approach

10 Phases 21 Days Estimated Status: Planning

Overview

This plan outlines the implementation approach for BodyBuddy v0.1, a minimal viable workout planning app for iOS. The implementation follows a bottom-up approach: build the core data models and workout engine first, then wrap them in SwiftUI views.

Strategy: Start with the foundation (models, engine, storage), then build up to the UI layer. This ensures the core logic is solid before adding presentation.

Implementation Phases

Phase 1 Foundation & Models

Days 1-2

Goal: Create the core Swift package with all data models and enums.

Tasks:
Create BodyBuddyCore Swift Package
Define all enums (PrimaryGoal, KneeProfile, MuscleGroup, EquipmentType, KneeLoadLevel, DayType)
Define core models (UserProfile, Exercise, WorkoutExercise, WorkoutSession, PreWorkoutCheckIn)
Add Identifiable and Codable conformance to all models
Write unit tests for model serialization/deserialization
Deliverable: Compilable BodyBuddyCore package with all models defined.
Dependencies: None

Phase 2 Exercise Library & Workout Engine

Days 3-4

Goal: Build the rules-based workout generation engine.

Tasks:
Create ExerciseLibrary.swift with static exercise database (15-20 exercises covering all muscle groups)
Tag exercises with kneeLoad (low/moderate/high), primaryMuscle, equipment, isCompound
Implement WorkoutEngine struct with generateWeek(for:startDate:) function
Implement exercise filtering logic based on knee profile
Implement day-type generation (fullBody only in v0.1)
Implement exercise selection algorithm (1 hinge, 1 push, 1 pull, 1-2 arms)
Implement adjust(session:with:) function for pre-workout check-ins
Write unit tests for engine logic (knee filtering, goal-based arm volume, check-in adjustments)
Deliverable: Fully functional workout engine that generates valid 3-day plans.
Phase 1 (models)

Phase 3 Data Persistence Layer

Day 5

Goal: Implement local storage for user profile and workout sessions.

Tasks:
Create DataStore protocol with methods: loadProfile(), saveProfile(), loadSessions(), saveSessions()
Implement JSONDataStore using FileManager to save/load JSON to Documents directory
Add error handling for file I/O operations
Write unit tests for persistence (save/load profile, save/load sessions)
Deliverable: Working local storage layer.
Phase 1 (models)

Phase 4 iOS App Structure & Navigation

Days 6-7

Goal: Set up the iOS app target with navigation and tab structure.

Tasks:
Create BodyBuddyApp iOS app target in Xcode
Link BodyBuddyCore package to app target
Create BodyBuddyApp.swift (main app entry point with @main)
Create RootView.swift with conditional logic: show onboarding if no profile, else show main tabs
Create empty placeholder views for: OnboardingView, TodayView, WorkoutPlayerView, SettingsView
Set up basic navigation structure (no functionality yet, just wireframes)
Deliverable: Navigable app shell with placeholder screens.
Phase 1 (models) Phase 3 (data store)

Phase 5 Onboarding Flow

Days 8-10

Goal: Build the complete onboarding experience.

Tasks:
Create OnboardingViewModel to manage state across steps
Implement GoalSelectionView (Step 1) with segmented control or cards for goal selection
Implement KneeProfileView (Step 2) with radio buttons for knee status
Implement ScheduleView (Step 3) with pickers for days/week and minutes/session
Implement EquipmentView (Step 4) with checkboxes for equipment types
Add navigation controls (Continue, Back buttons) between steps
On final step submit, save profile to data store and generate first week's workout plan
Trigger navigation to TodayView after onboarding completion
Deliverable: Fully functional onboarding flow that creates user profile and workout plan.
Phase 2 (workout engine) Phase 3 (data store) Phase 4 (navigation)

Phase 6 Today Screen & Pre-Workout Check-In

Days 11-12

Goal: Build the main dashboard and check-in feature.

Tasks:
Create TodayViewModel to load profile, sessions, and determine today's workout
Implement TodayView layout with three sections: check-in, workout summary, weekly overview
Implement PreWorkoutCheckInView with sliders for energy (1-5) and knee pain (0-2)
Implement "Adjust & Start Workout" button that calls WorkoutEngine.adjust() and navigates to Workout Player
Implement "Skip Check-in & Start" button that navigates directly to Workout Player
Display "Rest day" message when no workout is scheduled for today
Handle edge cases (no sessions generated, future dates)
Deliverable: Functional Today screen with working check-in adjustments.
Phase 2 (workout engine) Phase 3 (data store) Phase 5 (onboarding)

Phase 7 Workout Player

Days 13-15

Goal: Build the core workout tracking experience.

Tasks:
Create WorkoutViewModel to manage active session state
Implement WorkoutPlayerView layout: header with progress, current exercise card, set tracking table
Implement set completion logic (mark set as done, move to next set)
Implement exercise navigation (move forward/back through exercises)
Implement workout completion flow (mark session as complete, save to data store, navigate back to Today)
Implement progress persistence (save partial progress if user exits mid-workout)
Add simple elapsed timer display
Deliverable: Fully functional workout tracking with set completion and persistence.
Phase 3 (data store) Phase 6 (Today screen)

Phase 8 Settings Screen

Days 16-17

Goal: Allow users to view and edit their profile.

Tasks:
Create SettingsViewModel to load and update profile
Implement SettingsView layout with profile display and edit controls
Implement profile update logic (save changes to data store)
Implement plan regeneration (call workout engine with updated profile)
Implement data reset (delete all local files, show onboarding on next launch)
Deliverable: Functional settings screen with profile editing.
Phase 2 (workout engine) Phase 3 (data store)

Phase 9 Testing & Bug Fixes

Days 18-19

Goal: Comprehensive testing and bug fixing.

Tasks:
Manual testing of all user stories from spec.md
Test edge cases (no equipment selected, restricted knees with no low-load exercises, etc.)
Test data persistence (app restart, partial workout completion, profile changes)
Fix any bugs discovered during testing
Add logging/debugging statements for troubleshooting
Test on physical iOS device (not just simulator)
Deliverable: Stable, tested v0.1 app.
All previous phases

Phase 10 Polish & Documentation

Days 20-21

Goal: Final polish and documentation.

Tasks:
Add app icon and launch screen
Improve UI styling (colors, typography, spacing)
Add loading states and error messages
Write README.md with setup instructions
Write CHANGELOG.md
Create user-facing documentation (how to use the app)
Document code with comments where needed
Deliverable: Polished, documented v0.1 ready for personal use.
Phase 9 (testing complete)

Risk Assessment

High Risk

Data persistence edge cases

File I/O failures, corrupted JSON, app crashes mid-save

Mitigation: Comprehensive error handling, backup files, atomic writes

Medium Risk

Workout generation edge cases

No exercises match constraints

Mitigation: Fallback logic, warn user

UI/UX complexity

Set tracking, navigation, state management

Mitigation: Build incrementally, test frequently

Low Risk

Onboarding navigation

Relatively straightforward SwiftUI navigation

Mitigation: Use established SwiftUI patterns

Testing Strategy

Unit Tests (BodyBuddyCore):
  • Models: Test Codable conformance (encode/decode round-trip)
  • Workout Engine: Test exercise filtering, volume calculations, adjustment logic
  • Data Store: Test save/load operations, file path handling
Integration Tests:
  • Onboarding → Workout Generation: Verify profile saves and workout plan generates
  • Check-in → Adjustment: Verify check-in values correctly reduce sets
  • Workout Completion → Persistence: Verify completed sessions save and load
Manual Testing:
  • User Story Walkthroughs: Test each acceptance scenario from spec.md
  • Edge Cases: Test boundary conditions (no equipment, restricted knees, etc.)
  • Device Testing: Test on physical iPhone (not just simulator)

Success Metrics (v0.1)

  • User can complete onboarding and see a workout plan in under 3 minutes
  • Pre-workout check-in successfully adjusts workout volume
  • User can track and complete a full workout session
  • Workout plan respects knee constraints (no high-load exercises for sensitive/restricted)
  • Arm-focused users receive 2 arm exercises per session
  • App persists data across restarts (no data loss)
